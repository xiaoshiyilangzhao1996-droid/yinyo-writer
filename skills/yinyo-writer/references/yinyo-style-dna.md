# 写作风格 DNA

从 yinyo 已发布的写作实践中提取的完整写作风格规则。

---

## 一、句式指纹

### 1.1 平均句长
- 目标：15-25 字/句
- 单句不超过 25 字，用句号断句
- 长信息拆成多句，每句一个完整意思

**示例**：
```
✅ 好：用了 3 个月 Cursor 之后切换到 Claude Code。最大的感受是上下文理解能力完全不同。
   改一个功能，它知道要动哪几个文件。这个很关键。

❌ 差：在使用了大约 3 个月的 Cursor 之后，我开始转向使用 Claude Code，
   在这个过程中我最大的感受就是它在上下文理解方面有着完全不同的体验...
```

### 1.2 人称使用
- 第一人称高频出现："我"、"我的"、"我觉得"
- 频率目标：≥ 每千字 3 次"我"
- 用"我"的体验作为论据
- 对读者用"你"，建立对话感

### 1.3 确定性语气
- 有体验的 → 断言式表达（"这个很关键"）
- 没体验的 → 诚实标注（"我没用过，但从文档来看..."）
- 不做模糊中立的"两边都有道理"式判断

### 1.4 转折频率
- 转折词偏好：口语化的"但"、"不过"、"其实"
- 不用书面化的"然而"、"尽管"
- 转折频率：每千字 3-5 次

## 二、词汇特征

### 2.1 高频词
- "其实" — 引入真实感受或反转
- "你看" — 引导读者注意力
- "这里" — 指向具体细节
- "这个" — 替代抽象概念
- "关键" — 强调重要信息
- "说实话" / "讲真的" — 引入真实态度

### 2.2 具体数字偏好
- 用具体数字，不用模糊量词
  - ✅ "47 天里有 46 天" / "3 个月" / "最长跑了 1 天 18 小时 50 分钟"
  - ❌ "几乎每天" / "很长一段时间" / "跑了很久"
- 数字要可验证，不编造
- 数字放在有冲击力的位置

### 2.3 工具/产品名
- 用完整产品名："Claude Code"、"Cursor"、"GitHub Copilot"
- 不用泛称："某个 AI 工具"、"这类产品"
- 首次出现写全称，后可用简称

### 2.4 禁用词表

| 禁用 | 替代 |
|------|------|
| "接下来我们将进行..." | "我们来..." |
| "进行操作" | "点击" / "输入" |
| "实现功能" | "做到" / "搞定" |
| "综上所述" | 直接总结 |
| "值得注意的是" | 直接说 |
| "首先...其次...最后" | "第一" / "再说" / "还有一件事" |
| "强大的" / "革命性的" | 用具体事实和数据替代 |
| "可以说" | 直接说 |
| "一定程度上" | 直接说程度 |

## 三、节奏控制

### 3.1 开头节奏
- 快速进入主题，前 2-3 段内给出核心信息
- 不用"在当今这个 AI 时代"式的水开头
- 时间线锚点是首选："X 年前/月/天，我..."

### 3.2 主体节奏
- 表格/代码/文字交替
- 每 500-800 字一个视觉断点
- 类比级联：2-3 个递进类比解释一个概念

### 3.3 结尾节奏
- 每章结尾用"向前桥接"过渡
- 最后一章：开放式思考或行动召唤
- 结尾句要短（≤15 字），有余味

### 3.4 段落长度
- 正文段落 2-4 句
- 超过 4 句说明要拆分
- 代码块/表格前后的段落更要短（2 句以内）

## 四、核心写作技巧

### 4.1 时间线锚点

用具体时间线建立可信度。

**模式**：`[时间点/时间段] + [我做了什么] + [感受/结论]`

**示例**：
- "用了 3 个月 Cursor 之后切换到 Claude Code。最大的感受是..."
- "GA 后仅 6 个月，达到 10 亿美元年化收入。这个数字意味着什么？"
- "最长单次 session 跑了 1 天 18 小时 50 分钟。没崩过。"

### 4.2 类比级联

技术概念用 2-3 个递进类比解释。

**模式**：`A 是你的 X，B 是你的 Y，C 是你的 Z`

**示例**：
- "Copilot 是你的输入法，Cursor 是你的结对伙伴，Claude Code 是你的独立工程师"
- 递进逻辑：被动辅助 → 主动协作 → 独立执行

### 4.3 角色定义

告诉读者"你是谁"。

**模式**：`"从现在开始，你是 [角色]"`

**示例**：
- ✅ "从现在开始，你是产品经理，Claude 是你的工程师"
- ✅ "你是给指令的人，不是写代码的人"
- ❌ "你应该学会管理 AI 工具"

### 4.4 数据解读

呈现数据后，必须给一句话解读。

**模式**：`"重点看 [位置]。" + "这些数字背后的信号就一个：[结论]"`

## 五、内容模块格式

### 5.1 对比表格

```markdown
| 维度 | 产品A | 产品B | 结论 |
|------|-------|-------|------|
| 价格 | $20/月 | $100/月 | 个人选 A |
| 速度 | 快 | 慢 | A 的优势 |
```

**规则**：
- 必须有"结论"列
- 表格后必须有解读段落
- 维度要具体

### 5.2 代码块

````markdown
```python
# 这行是关键：设置 API Key
import os
os.environ["API_KEY"] = "your-key"  # 替换为你的 Key
```

这里要注意，[解释为什么]。
````

**规则**：
- 带语言标签
- 关键行加行内注释
- 代码块前后有说明文字

### 5.3 步骤说明

```markdown
第一步：[动作]
```
[命令/截图/操作]
```
预期结果：[你看到了什么]
```

**规则**：
- 每步一个动作，不合并
- 每步有预期结果
- 踩坑点单独标注

## 六、风格标尺

在写作完成后，用以下标尺自检：

| 维度 | 口语 ←→ 书面 | 目标位置 |
|------|-------------|---------|
| 整体语气 | 偏向口语 | 左侧 70-80% |
| 抽象 vs 具体 | 偏向具体 | 右侧 70-80% |
| 谨慎 vs 断言 | 有体验就断言 | 动态 |
| 学术 vs 通俗 | 偏向通俗 | 右侧 80-90% |
| 长句 vs 短句 | 偏向短句 | 右侧 80-90% |
| 铺垫 vs 结论先行 | 结论先行 | 右侧 70-80% |

## 七、不同文章类型的风格微调

### 深度评测型
- 表格占比 30%+
- 结论前置
- 评测维度要有个人标准

### 实战教程型
- 代码块占比 30%+
- 每步有截图位置标注
- 踩坑经验单独标注

### 洞察观点型
- 类比和比喻高频
- 个人经历作为论据
- 要有可操作的结论

## 八、书籍专用规则

### 8.1 双语副标题

从入门到精通类型使用英文副标题：

```markdown
# Claude Code 从入门到精通

A Complete Guide from Beginner to Master
```

### 8.2 阅读指南格式

```markdown
### 阅读指南

| 时间 | 章节 | 目标 |
|------|------|------|
| Day 1 | §01-§03 | 从零到第一次跑通 |
| Day 2-3 | §04-§06 | 掌握核心能力 |
| Day 4-5 | §07-§10 | 进阶实战 |
```

### 8.3 跨章一致性

- 每章开头模式必须一致
- 每章结尾必须有向前桥接
- 内容块格式全书统一

### 8.4 章节交叉引用

```markdown
关于安装细节，参见 §02。
配置部分我们在 §08 详细讲。
```

**规则**：
- 用 `§NN` 格式引用
- 只在必要时引用
