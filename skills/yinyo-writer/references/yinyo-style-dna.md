# 写作风格 DNA

从 yinyo 已发布的写作实践中提取的完整写作风格规则。

---

## 一、句式指纹

### 1.1 平均句长
- 第一性原理：句子的目标是承载一个清楚的语义单元，15-25 字只是体感参考，不是硬指标。
- 同一组并列、递进、解释或转折信息，优先用逗号、分号、冒号或破折号承接；不要为了短句机械使用句号。
- 短句只用于强调、转折和落点；避免连续 3 句以上只靠句号推进的问答、判断或解释。

**示例**：
```
✅ 好：用了 3 个月 Cursor 后，我切到 Claude Code；最大的感受是上下文理解完全不同：
   改一个功能，它知道要动哪几个文件，这很关键。

❌ 差：用了 3 个月 Cursor。然后我切到 Claude Code。最大的感受是上下文理解不同。
   改一个功能。它知道要动哪几个文件。这个很关键。
```

### 1.2 人称使用
- 第一人称高频出现："我"、"我的"、"我觉得"
- 频率目标：≥ 每千字 3 次"我"
- 用"我"的体验作为论据
- 对读者用"你"，建立对话感

### 1.3 确定性语气
- 有体验的 → 断言式表达（"这个很关键"）
- 没体验的 → 诚实标注（"我没用过，但从文档来看..."）
- 不做模糊中立的"两边都有道理"式判断

### 1.4 转折频率
- 转折词偏好：口语化的"但"、"不过"、"其实"
- 不用书面化的"然而"、"尽管"
- 转折频率：每千字 3-5 次

### 1.5 历史通俗叙事感
- 第一性原理：好读的技术文章要先识别概念背后的冲突，再安排概念登场。
- 技术路线要人物化：RAG、SAG、GraphRAG、LLM Wiki 可以像角色一样登场，但不要写成幼稚拟人剧。
- 每个核心概念至少经历一次"旧秩序 -> 裂缝 -> 新角色登场 -> 代价和边界"的推进。
- 夹叙夹议：讲一段机制、补一句判断；讲一个场景、点出一个后果。不要整段堆设定，也不要整段喊观点。
- 幽默要干：用克制吐槽、历史感比喻和反差收束，不讲段子，不抖包袱，不为了好笑牺牲判断。
- 禁止精确模仿任何在世作者的专属措辞、标志性节奏或可识别段落；只使用"通俗历史叙事"的通用特征。

**示例**：
```
✅ 好：RAG 刚登场时，很多团队以为知识库终于有了答案。后来企业数据一上来，事情开始变味：
   资料能找回，关系却连不上；答案能生成，知识却留不住。于是 SAG 和 LLM Wiki 分别从两条路进场。

❌ 差：RAG 是一种检索增强生成方法。SAG 是一种新的 RAG 方法。LLM Wiki 是一种知识库维护方法。
   它们各有优势。下面开始分析。
```

## 二、词汇特征

### 2.1 高频词
- "其实" — 引入真实感受或反转
- "你看" — 引导读者注意力
- "这里" — 指向具体细节
- "这个" — 替代抽象概念
- "关键" — 强调重要信息
- "说实话" / "讲真的" — 引入真实态度

### 2.2 具体数字偏好
- 用具体数字，不用模糊量词
  - ✅ "47 天里有 46 天" / "3 个月" / "最长跑了 1 天 18 小时 50 分钟"
  - ❌ "几乎每天" / "很长一段时间" / "跑了很久"
- 数字要可验证，不编造
- 数字放在有冲击力的位置

### 2.3 工具/产品名
- 用完整产品名："Claude Code"、"Cursor"、"GitHub Copilot"
- 不用泛称："某个 AI 工具"、"这类产品"
- 首次出现写全称，后可用简称

### 2.4 禁用词来源

禁用词和替代表达统一以 `references/banned-words.md` 为准。本文档只定义风格指纹，不维护第二份禁用词表。

## 三、节奏控制

### 3.1 开头节奏
- 快速进入主题，前 2-3 段内给出核心信息
- 不用"在当今这个 AI 时代"式的水开头
- 时间线锚点是首选："X 年前/月/天，我..."

### 3.2 主体节奏
- 表格/代码/文字交替
- 每 500-800 字一个视觉断点
- 类比级联：2-3 个递进类比解释一个概念

### 3.3 结尾节奏
- 每章结尾用"向前桥接"过渡
- 最后一章默认使用"宏大本质 -> 哲学反问 -> 人类微光"结构，尤其适用于 AI、Agent、知识库、自动化、模型能力这类主题。
- 宏大本质：先说清 AI 现象背后的尺度变化，例如参数、自动化、知识压缩、组织记忆、机器推理。
- 哲学反问：打破常规认知，追问"如果机器越来越会做这件事，人到底还剩下什么不可替代"；不要写成连续问句审问读者。
- 人类微光：回到一个具体、微小、坚固的人类特质，例如心跳、犹豫、羞耻、疼痛、牵挂、责任、记忆里的温度。
- 结尾句要短（≤15 字），有余味；余味来自具体的人类细节，不来自空泛升华。
- 避免用正反对照套句收束；改成正向判断、哲学追问或具象画面。

**示例**：
```
✅ 好：当 Agent 开始替我们检索、总结、更新知识，组织记忆第一次有了机器速度。
   但如果一切经验都能被压缩成索引和页面，人还剩下什么需要亲自记住？
   也许答案藏在很小的地方：一次迟疑、一次确认、一次不愿把责任交出去的停顿。
   那一秒，人才重新出现。

❌ 差：AI 知识库很重要。它改变了知识管理。结尾只剩口号，没有具体的人。
```

### 3.4 段落长度
- 正文段落 2-4 句
- 超过 4 句说明要拆分
- 代码块/表格前后的段落更要短（2 句以内）
- 不要把同一个语义组拆成连续句号堆叠；3 个以上并列判断共享同一框架时，用冒号引出、分号并列，或改成表格

## 四、核心写作技巧

### 4.1 时间线锚点

用具体时间线建立可信度。

**模式**：`[时间点/时间段] + [我做了什么] + [感受/结论]`

**示例**：
- "用了 3 个月 Cursor 之后切换到 Claude Code。最大的感受是..."
- "GA 后仅 6 个月，达到 10 亿美元年化收入。这个数字意味着什么？"
- "最长单次 session 跑了 1 天 18 小时 50 分钟。没崩过。"

### 4.2 类比级联

技术概念用 2-3 个递进类比解释。

**模式**：`A 是你的 X，B 是你的 Y，C 是你的 Z`

**示例**：
- "Copilot 是你的输入法，Cursor 是你的结对伙伴，Claude Code 是你的独立工程师"
- 递进逻辑：被动辅助 → 主动协作 → 独立执行

### 4.3 角色定义

告诉读者"你是谁"。

**模式**：`"从现在开始，你是 [角色]"`

**示例**：
- ✅ "从现在开始，你是产品经理，Claude 是你的工程师"
- ✅ "你是给指令的人，不是写代码的人"
- ❌ "你应该学会管理 AI 工具"

### 4.4 数据解读

呈现数据后，必须给一句话解读。

**模式**：`"重点看 [位置]。" + "这些数字背后的信号就一个：[结论]"`

## 五、内容模块格式

### 5.1 对比表格

```markdown
| 维度 | 产品A | 产品B | 结论 |
|------|-------|-------|------|
| 价格 | $20/月 | $100/月 | 个人选 A |
| 速度 | 快 | 慢 | A 的优势 |
```

**规则**：
- 必须有"结论"列
- 表格后必须有解读段落
- 维度要具体

### 5.2 代码块

````markdown
```python
# 这行是关键：设置 API Key
import os
os.environ["API_KEY"] = "your-key"  # 替换为你的 Key
```

这里要注意，[解释为什么]。
````

**规则**：
- 带语言标签
- 关键行加行内注释
- 代码块前后有说明文字

### 5.3 步骤说明

```markdown
第一步：[动作]
```
[命令/截图/操作]
```
预期结果：[你看到了什么]
```

**规则**：
- 每步一个动作，不合并
- 每步有预期结果
- 踩坑点单独标注

## 六、风格标尺

在写作完成后，用以下标尺自检：

| 维度 | 口语 ←→ 书面 | 目标位置 |
|------|-------------|---------|
| 整体语气 | 偏向口语 | 左侧 70-80% |
| 抽象 vs 具体 | 偏向具体 | 右侧 70-80% |
| 谨慎 vs 断言 | 有体验就断言 | 动态 |
| 学术 vs 通俗 | 偏向通俗 | 右侧 80-90% |
| 长句 vs 短句 | 服从语义节奏，落点可短 | 动态 |
| 铺垫 vs 结论先行 | 结论先行 | 右侧 70-80% |

## 七、不同文章类型的风格微调

### 深度评测型
- 表格占比 30%+
- 结论前置
- 评测维度要有个人标准

### 实战教程型
- 代码块占比 30%+
- 每步有截图位置标注
- 踩坑经验单独标注

### 洞察观点型
- 类比和比喻高频
- 个人经历作为论据
- 要有可操作的结论

## 八、书籍专用规则

### 8.1 双语副标题

从入门到精通类型使用英文副标题：

```markdown
# Claude Code 从入门到精通

A Complete Guide from Beginner to Master
```

### 8.2 阅读指南格式

```markdown
### 阅读指南

| 时间 | 章节 | 目标 |
|------|------|------|
| Day 1 | §01-§03 | 从零到第一次跑通 |
| Day 2-3 | §04-§06 | 掌握核心能力 |
| Day 4-5 | §07-§10 | 进阶实战 |
```

### 8.3 跨章一致性

- 每章开头模式必须一致
- 每章结尾必须有向前桥接
- 内容块格式全书统一

### 8.4 章节交叉引用

```markdown
关于安装细节，参见 §02。
配置部分我们在 §08 详细讲。
```

**规则**：
- 用 `§NN` 格式引用
- 只在必要时引用
