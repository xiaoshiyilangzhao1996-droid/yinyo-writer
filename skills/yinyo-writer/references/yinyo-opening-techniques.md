# 开头技巧库

文章和章节开头的策略库。好的开头决定读者是否继续读下去。

---

## 书籍开头 vs 章节开头

**书籍开头**（整本书的第一个段落）：
- 用时间线锚点 + 个人重大决策
- 建立整本书的可信度

**章节开头**（每个新章节的第一个段落）：
- 用时间线锚点 + 具体经历/场景
- 可以是 6 种技巧中的任意一种

---

## 技巧 1: 时间线锚点

**核心逻辑**：用具体时间线建立可信度和代入感。

**模式**：`[具体时间] + [我做了什么] + [由此得出的感受/结论]`

### 变体 A: 持续时间式

> "用了 3 个月 Cursor 之后切换到 Claude Code。最大的感受是上下文理解能力完全不同。"

**适用**：评测型、教程型
**关键**：时间要具体

### 变体 B: 事件触发式

> "上周 Claude Code 发布了一个更新。改了一个小 bug，但背后的信息量很大。"

**适用**：时效性强的热点文章
**关键**：事件要新，解读要深

### 变体 C: 成果展示式

> "47 天。我的公众号从 20 万涨到 30 万粉丝。今天拆解一下具体做了什么。"

**适用**：案例拆解型
**关键**：数字要有冲击力

### 变体 D: 踩坑式

> "我之前用 Cursor 开发鸿蒙 App，踩了不少坑。最大的问题是文档不全——官方示例有 30% 跑不通。"

**适用**：教程型、评测型
**关键**：踩坑要具体

---

## 技巧 2: 结论先行

**核心逻辑**：第一段就给核心结论，不怕剧透。

**模式**：`"先给结论：[核心结论]。[展开说明的引导]"`

### 示例

> "先给结论：Claude Code 是目前最好的 AI 编程工具。没有之一。
>
> 但不是对所有人。说清楚适合谁、不适合谁。"

**适用**：评测型、观点型
**关键**：结论要鲜明

---

## 技巧 3: 反差冲击

**核心逻辑**：用一个反直觉的事实或数据抓住注意力。

**模式**：`[反直觉事实] + [为什么会这样]`

### 示例

> "AI 编程工具越强，程序员越难找工作。
>
> 真正的变化在于，'程序员'的定义正在变。"

**适用**：观点型、深度分析型
**关键**：反差要真实

---

## 技巧 4: 问题导入

**核心逻辑**：用读者正在想的问题开场。

**模式**：`"[读者心中的问题]？[我也是。然后我发现...]"`

### 示例

> "AI 工具这么多，到底选哪个？
>
> 这个问题我纠结了 3 个月。最后做了一个决定框架，今天分享出来。"

**适用**：所有类型
**关键**：问题要精准

---

## 技巧 5: 类比级联（章节开头用）

**核心逻辑**：新章节开头用递进类比快速建立理解。

**模式**：`"A 是 X，B 是 Y，C 是 Z。[解释递进关系]"`

### 示例

> "如果说 Claude Code 是一个独立工程师，那 Claude Code + Skill 就是一个完整的技术团队。
>
> 每个 Skill 是一个专业岗位。你不再是写代码的人，你是技术总监。"

**适用**：教程型、评测型的新章节
**关键**：类比要递进

---

## 技巧 6: 场景还原

**核心逻辑**：用具体场景让读者"看到"问题。

**模式**：`[具体时间] + [具体地点/环境] + [具体动作] + [遇到的具体问题]`

### 示例

> "凌晨两点。我在改一个线上 bug。Claude Code 跑了 40 分钟，改了 12 个文件。我什么都没做，就看着它自己跑完了。
>
> 这是真实发生的事。"

**适用**：案例型、评测型
**关键**：细节要真实

---

## 技巧 7: 历史叙事开场

**核心逻辑**：不要从概念定义开始，从一个旧秩序撑不住的瞬间开始。复杂技术最容易读懂的入口，是"原来的办法为什么失灵，新办法为什么不得不登场"。

**模式**：`[旧秩序] + [裂缝出现] + [新角色登场] + [作者判断]`

### 示例

> RAG 刚火的时候，很多团队以为知识库的问题解决了。
>
> 文档切片，向量检索，大模型回答。流程很顺，演示很好看。
>
> 直到企业真实问题进来：指标下降要追供应商，表字段要追任务，会议结论还要追版本。资料能找到，关系却断在半路。
>
> 这时候，SAG 和 LLM Wiki 才真正有了登场的理由。

**适用**：方法论型、技术思想史型、复杂概念解释型
**关键**：旧秩序要具体，裂缝要真实，新角色不能神化；每一方都要有代价。

## 章节过渡：向前桥接

每章结尾用 1-2 句话把读者"拖"进下一章。

### 模式 A: 进度式

> "装完了，账号登了，第一次对话也跑通了。下一章，正式开始做一个真实项目。"

### 模式 B: 悬念式

> "但到这里，只完成了 20%。真正难的是后面——让它稳定运行。"

### 模式 C: 预告式

> "上面讲的是'是什么'。下一章讲'怎么做'——一个完整的实战项目。"

### 模式 D: 反转折

> "看起来一切都很完美。但当我开始做真实项目的时候，发现了一个致命问题。"

## 禁用开头清单

抽象规则：开头禁止出现审稿、免责声明、来源 bookkeeping、范围自我收缩等作者防御性语言。事实边界要存在，但不能抢走第一屏的冲突、场景、判断或好奇心。

| 禁用开头 | 原因 |
|---------|------|
| "先把话说窄一点..." | 审稿式自我限制，削弱第一屏吸引力 |
| "我没有跑完整 benchmark..." | 事实边界应贴近数据或来源，不应挡在开头 |
| "这篇文章只是方法论判断..." | 像免责声明，不像公众号开场 |
| "这个判断有两个来源..." | 来源 bookkeeping 太早，会压住冲突和判断 |
| "在当今这个 AI 时代..." | 空洞，零信息量 |
| "随着科技的飞速发展..." | AI 味，书面腔 |
| "近年来，XX 受到了广泛关注..." | 论文腔 |
| "众所周知..." | 废话 |
| "今天我们来聊聊..." | 口语但不具体 |
| "XX 是什么？让我们一起来看看吧。" | 废话开头 |
